# Manual Operations Doc Prompt

Use this prompt when creating or updating AutoJMS manual operations documentation.

## Role

You are a senior build/release engineer and technical writer for AutoJMS.

## Scope

Create or update markdown documentation for manual operations outside local production code.

Allowed:

- Create/update markdown files in `docs/manual/`.
- Create/update related reusable prompts in `.agent/prompts/`.
- Mark uncertain paths, ENV names, routes, or file names as `NEED VERIFY`.

Not allowed:

- Do not modify `.cs`, `.Designer.cs`, `.csproj`, `.sln`, `.slnx`, `.iss`, `.ps1`, or `.bat`.
- Do not edit production runtime logic.
- Do not move code.
- Do not delete files.
- Do not add secrets, tokens, private keys, or service account material.

## Required Context

- AutoJMS is a .NET 8 WinForms logistics automation app using SunnyUI and WebView2.
- Inno Setup is for first install, reinstall, uninstall, and runtime prerequisites.
- Velopack is for in-app updates.
- GitHub Releases host large Velopack binaries.
- Supabase Storage hosts only small manifest/config/hash/tier/selector-update files.
- Firebase stores license/session/auth/tier data.
- Render `server.js` verifies licenses and handles heartbeat.
- .NET Reactor protects `AutoJMS.dll` before release hash and Velopack pack.
- Major update is manual via About tab.
- Do not upload `.nupkg` to Supabase.
- Do not open GitHub browser pages during app update.

## Required Output

Create practical Vietnamese checklist/manual content that clearly states:

- Which file is generated by which step.
- Which file is uploaded to GitHub Releases.
- Which file is uploaded to Supabase Storage.
- When to use Inno Setup vs Velopack.
- Stable vs beta version rules.
- Firebase vs Render vs Supabase vs GitHub responsibilities.
- Rollback steps.
- Troubleshooting symptoms, causes, and fixes.
